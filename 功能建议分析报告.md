# 妮妮一日三餐 · 项目现状分析与功能建议

> 评估时间：2026-07-14 | 评估角色：资深体验用户

---

## 一、项目现状评价

### 整体评价：完成度 7.5/10

这是一个**情侣美食点餐小程序**，核心概念清晰——女朋友点菜、男朋友做菜，妮妮币作为虚拟货币串联全流程。从产品角度看，核心闭环已经跑通：

```
注册登录 → 浏览菜品 → 加购物车 → 下单扣币 → 男友接单 → 制作完成
```

### 做得好的地方

1. **角色权限设计精准** — 男朋友看不到点餐 Tab、不能下单；女朋友不能操作订单状态。权限隔离干净利落。
2. **妮妮币体系完整** — 注册送 1000 币、下单扣款、拒绝退款、充值（男友专属），经济系统闭环。
3. **订单状态机严谨** — pending → accepted → preparing → completed 五状态流转，拒绝自动退款，状态转换有校验。
4. **自定义分类/标签** — 用户可以输入任意分类和口味标签，灵活度高。
5. **旧数据兼容** — openId/openid 大小写迁移，说明有版本迭代意识。

### 当前体验痛点

1. **下单后陷入"盲等"** — 女朋友下单后只能手动刷新订单页看状态变化，不知道男朋友是否看到了
2. **做完菜没有"仪式感"** — 男朋友标记完成后，女朋友只看到状态变为"已完成"，看不到成品照片
3. **菜品一多就难找** — 没有搜索功能，分类 Tab 只能筛选不能搜索
4. **首页"最近吃过"是空的** — 有占位但没数据，每次看到都是"还没有记录哦"
5. **男友无法管理菜品** — 菜品只能女朋友添加，男友不能编辑价格、设为推荐、删除过时的菜
6. **没有消费数据反馈** — 用了多少妮妮币、最常吃什么、这个月做了几次饭，一概不知

## 二、新功能建议（按优先级分层）

### P0 · 体验痛点修复（建议优先做）

#### 1. 订单状态通知

**痛点**：女朋友下单后只能不断刷新页面看状态，男朋友也可能漏掉新订单。

**方案**：使用微信小程序的**订阅消息**能力，在以下节点推送通知：
- 女朋友下单 → 男朋友收到"新订单来了"通知
- 男朋友接单/开始制作 → 女朋友收到"男朋友已接单/开始制作"通知
- 男朋友标记完成 → 女朋友收到"饭好啦，快来吃"通知
- 男朋友拒绝 → 女朋友收到"订单被拒，妮妮币已退回"通知

**实现要点**：
- 创建一个 `sendSubscribeMessage` 云函数
- 下单时让女朋友授权订阅消息（`wx.requestSubscribeMessage`）
- `updateOrderStatus` 云函数状态变更时同时发推送
- 需要在微信公众平台配置消息模板

#### 2. 成品照片上传

**痛点**：男朋友做完菜标记完成后，女朋友只看到状态变化，没有任何"成果展示"，缺少互动仪式感。

**方案**：在男朋友点击"制作完成"时，弹窗让他上传 1-3 张成品照片，写入订单的 `completionImages` 字段。女朋友在订单页看到已完成订单时，可以查看成品照片。

**实现要点**：
- orders 集合新增 `completionImages: []` 字段
- `updateOrderStatus` 云函数在状态变为 completed 时接收图片 fileID
- 订单卡片增加成品照片展示区（轮播或九宫格）

#### 3. 订单评价/反馈

**痛点**：吃完饭没有任何反馈闭环，男朋友不知道做得好不好吃，女朋友也没地方表达。

**方案**：已完成订单增加评价功能：
- 5 星评分（爱心打分，呼应现有 UI 风格）
- 文字评价（可选，最多 100 字）
- "下次还想吃" 按钮（一键重新点这道菜）

**实现要点**：
- orders 集合新增 `rating`、`review`、`wantAgain` 字段
- 创建 `submitReview` 云函数
- 订单页已完成状态显示"去评价"或"已评价"标签
- 首页可以展示"妮妮好评榜"

#### 4. 首页"最近吃过"填充

**痛点**：首页有一个"最近吃过"区域，但永远是空的"还没有记录哦"，每次打开都觉得这个 App 没人在用。

**方案**：查询最近 5 个已完成订单的菜品，在首页展示。展示形式：横向滚动的菜品卡片（图片 + 名称 + 上次吃的时间）。

**实现要点**：
- `getOrders` 云函数增加 `recentCompleted` 查询模式，返回最近完成的订单菜品
- 首页 `fetchRecentFoods` 方法调用并渲染
- 点击卡片可以直接重新加入购物车

#### 5. 订单内留言/沟通

**痛点**：女朋友下单时虽然有"备注"字段，但下单后的沟通完全没有。比如"多放点辣"、"其实不要葱了"这种需求没法表达。

**方案**：在订单详情中增加简易留言板：
- 女朋友和男朋友都可以发文字消息
- 消息附带发送时间和发送人
- 类似简易聊天，但只限当前订单内

**实现要点**：
- orders 集合新增 `messages: [{ from, text, createdAt }]` 数组字段
- 创建 `addOrderMessage` 云函数
- 订单卡片展开后显示留言列表 + 输入框
- 轮询或 `wx.connectSocket` 实时刷新

#### 6. 预计完成时间

**痛点**：男朋友接单后，女朋友不知道要等多久，只能干等着。

**方案**：男朋友接单时可以选择预计完成时间（15分钟/30分钟/45分钟/1小时/自定义），女朋友在订单页看到倒计时。

**实现要点**：
- orders 集合新增 `estimatedMinutes` 和 `acceptedAt` 字段
- 接单弹窗增加时间选择器
- 女朋友侧根据 `acceptedAt + estimatedMinutes` 计算剩余时间，显示倒计时
- 超时可以给男朋友提示

### P1 · 功能增强（建议第二批）

#### 7. 菜品搜索

**痛点**：菜品积累到 20+ 后，分类筛选不够用，想找一道特定的菜要翻很久。

**方案**：点餐页顶部增加搜索栏，支持按名称、分类、口味标签模糊搜索。

**实现要点**：
- `getFoods` 云函数增加 `keyword` 参数，后端用正则匹配
- 搜索栏使用 `input` 组件 + 防抖（300ms）
- 搜索结果高亮匹配关键词
- 增加热门搜索标签（基于下单频率）

#### 8. 菜品详情页

**痛点**：目前菜品只是一个卡片，看不到大图、看不到历史评价、看不到之前做过的成品照片。

**方案**：新增 `pages/food-detail/index` 页面：
- 顶部大图轮播（菜品图片 + 历史成品照片）
- 菜品信息（名称、分类、价格、口味标签、想吃程度）
- 历史评价列表（从已完成订单的评价中聚合）
- 历史下单记录（上次吃是什么时候）
- 底部"加入购物车"按钮

#### 9. 收藏/心愿单

**痛点**：每次要点常吃的菜都要翻分类找，没有快速入口。

**方案**：
- 菜品卡片增加收藏按钮（爱心图标）
- 点餐页增加"收藏"分类 Tab
- foods 集合在用户维度记录收藏状态（新建 `favorites` 集合或 users 集合增加 `favorites: []`）

#### 10. 数据统计面板

**痛点**：用了一段时间后，完全不知道自己的消费情况，妮妮币花了多少、最常吃什么。

**方案**："我的"页增加统计卡片，点击进入统计详情页：
- 本月消费妮妮币总额
- 累计点单次数 / 完成次数
- 最常吃的 Top 3 菜品
- 连续做饭天数
- 简单的柱状图展示近 7 天下单趋势

**实现要点**：
- 创建 `getStatistics` 云函数聚合订单数据
- 首页"我的"页展示概览数字
- 新增统计详情页用 Canvas 或 ECharts 画图

#### 11. 菜品管理（男友端）

**痛点**：菜品只能女朋友添加，男朋友不能编辑价格、删除过时的菜、设置推荐。

**方案**：
- 男朋友的首页增加"菜品管理"入口
- 菜品列表支持编辑（名称、价格、分类、图片）和删除
- 支持"设为推荐"/"取消推荐"开关
- 创建 `updateFood` 和 `deleteFood` 云函数

#### 12. 烹饪笔记

**痛点**：男朋友做了几次同一种菜，每次都要重新想做法，没有地方记录。

**方案**：菜品详情页增加"烹饪笔记"区域，男朋友可以记录：
- 做法步骤（文字）
- 用了什么食材
- 改进点（"下次少放盐"）
- 笔记关联菜品 ID，按菜品维度展示

### P2 · 锦上添花（有空再做）

#### 13. 成就/打卡系统

- 连续做饭 7 天 / 30 天 / 100 天徽章
- 累计做饭 50 道菜勋章
- "五星大厨" — 获得 10 个五星评价
- 首页展示当前成就进度

#### 14. 周菜单规划

- 首页增加"本周菜单"区域
- 可以提前为周一到周日各安排一道菜
- 到了那天自动提醒"今天该做 XX 了"
- 支持从收藏/历史中快速选菜

#### 15. 纪念日特别菜单

- 个人中心设置生日、在一起纪念日等特殊日期
- 到了纪念日首页自动切换主题色 + 特别推荐
- "周年纪念套餐"推荐

#### 16. 食材购物清单

- 男朋友接单后，自动生成食材清单
- 支持手动添加食材项 + 打勾标记已买
- 清单可以分享到微信聊天

#### 17. 菜品分享卡片

- 菜品详情页"分享"按钮
- 生成精美卡片（菜品图 + 名称 + 评价）
- 分享到微信聊天或朋友圈

#### 18. 随机点菜（选择困难症救星）

- 点餐页增加"帮我选"按钮
- 从所有菜品中随机推荐一道
- 可以加筛选条件（分类、价格区间）
- 摇一摇触发随机选菜

## 三、技术债务提醒

| 问题 | 建议 |
|------|------|
| `pages/logs/` 残留空壳文件 | 清理删除，避免混淆 |
| `utils/util.js` 只有 1 个函数 | 随着功能增加建议拆分工具类 |
| 购物车数据只在 Storage | 换手机会丢失，建议同步到云数据库 |
| 菜品 `isRecommended` 需手动设置 | 结合功能 #11 让男友可以直接操作 |
| 口味标签不自定义不同步 | 建议独立 `tags` 集合，全局复用 |
| 无加载失败重试机制 | 网络异常时建议增加重试 + 空状态引导 |

## 四、建议实施顺序

```
第一批 (1-2周)：通知推送 + 成品照片 + 首页最近吃过
         → 解决最痛的体验问题，让核心闭环完整

第二批 (2-3周)：订单评价 + 订单留言 + 预计完成时间
         → 补全互动闭环，增加情感连接

第三批 (2-3周)：菜品搜索 + 菜品详情页 + 菜品管理
         → 提升使用深度，菜品多了也好用

第四批 (2周)：数据统计 + 收藏
         → 增加留存抓手

第五批 (有空就做)：成就系统 + 周菜单 + 纪念日 + 随机点菜
         → 锦上添花，让产品更有温度
```
